# 网络设备采集分析平台 — 全局配置方案

## 1. 整体设计

全局配置是工具首次使用时的初始化数据，包含两部分：
- SSH 连接配置：定义如何登录堡垒机
- Region+Section 映射：定义从(Region, Section)到SSH连接的关联规则

存储方式：本地 SQLite + JSON 导入导出。

## 2. SSH 连接配置

每个 SSH 连接包含以下字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| name | string | Y | 连接名称，唯一标识 |
| host | string | Y | 堡垒机域名或IP |
| port | int | Y | SSH端口，默认22 |
| username | string | Y | 登录用户名 |
| key_path | string | Y | SSH私钥文件路径 |
| key_password | string | N | 私钥密码（可选） |
| expect_flow | array | Y | 登录后的expect/send流程 |

expect_flow 定义格式：一组 (expect匹配文本, send发送内容) 的有序列表
其中 {device_ip} 为运行时变量，采集时自动替换为目标设备IP。

## 3. Region+Section 映射

### 设计原则
- Region + Section → SSH 连接
- 支持默认连接(deaccessult) + 特例覆盖(overrides)

### 数据结构

```
{
  "RegionB": {
    "deaccessult": "大云堡垒机",
    "overrides": {}
  },
  "RegionA": {
    "deaccessult": "大云堡垒机",
    "overrides": {
      "Rack1-%": "苏州堡垒机"
    }
  }
}
```

### 解析逻辑

```
def resolve_ssh(region, section):
    mapping = config["region_mapping"].get(region, {})
    if section in mapping.get("overrides", {}):
        return mapping["overrides"][section]
    return mapping.get("deaccessult")
```

## 4. 导入导出

- 导出格式：单个 JSON 文件
- 导出内容：SSH 连接配置（password清空）+ Region+Section映射
- 导入：解析 JSON -> 写入 SQLite（password字段留空，需用户补填）
- 私钥密码仅在运行时内存中存在，不持久化到导出文件

## 5. 全局配置模板完整示例

包含：2个SSH连接 + mapping规则，直接在GUI或JSON中编辑。

## 6. 其他

- 全局配置在首次运行引导界面完成
- 支持修改和重新导入
- SSH连接密码落库时建议用 AES 加密存储，密钥来自用户设定的主密码

# 设备管理模块

## 设备来源
- 核心设备：Excel 批量导入（变动少，一次性导入）
- 接入设备：GUI 手工录入（变动频繁）

## 设备属性
| 字段 | 说明 |
|------|------|
| hostname | 设备名 |
| ip | 管理IP |
| region | 区域 |
| section | 设备归属路径，多级用 / 分隔，如 Rack1-Core-01。支持 LIKE 通配符匹配 |
| role | 角色（核心/汇聚/接入/出口/FW...） |
| vendor | 厂商（VendorA/VendorB） |
| description | 备注 |

设备唯一键：region + section + hostname（IP 可能变更）

## Excel 导入
模板格式（CSV）：
```
hostname,ip,region,section,role,vendor,description
Rack1-Core-01,192.168.1.1,RegionA,Rack1-Core,core,VendorA,核心
Rack1-Access-01,192.168.2.1,RegionA,Rack1-Access-01,distribution,VendorB,汇聚
```

导入策略：
- 首次：全量导入
- 再次导入：同名设备覆盖更新，新设备追加，不在 Excel 里的不删除
- 导入前预校验：必填字段检查、格式检查、显示变更预览

## 手工录入
GUI 表单：填写 7 个字段 → 即时写入 SQLite

## 数据表
```sql
CREATE TABLE devices (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    hostname TEXT NOT NULL,
    ip TEXT NOT NULL,
    region TEXT NOT NULL,
    section TEXT NOT NULL,
    role TEXT NOT NULL,
    vendor TEXT,
    description TEXT,
    created_at TEXT,
    updated_at TEXT,
    UNIQUE(region, section, hostname)
);
```

## 任务匹配设备
创建任务时：
1. 场景模板定义（section+role 通配符）
2. 用户选 Region
3. 系统自动从 devices 表筛选：
   ```
   SELECT * FROM devices 
   WHERE region = '{选中的Region}' 
     AND section GLOB '{section通配符}'
     AND role = '{role}'
   ```
4. 展示匹配结果列表，用户可勾选/增删

> **通配符匹配优先级**：精确匹配优先于通配符。匹配顺序为：具体 section 名 > glob 模式 > deaccessult。

# 任务模块设计

任务 = 一次完整的采集分析工作流。

## 任务创建流程

### 1. 选择场景模板
从已保存的场景列表中选一个（含版本），加载场景定义。

### 2. 填写场景输入参数
场景定义中声明的输入参数，在此填写：
- 待放通网段：10.0.0.0/24

### 3. 选择 Region
决定从哪个区域采集。

- **采集超时**（每台设备超时秒数，默认 60s）
- **任务名称**（自动生成：场景名_日期_序号，也可手动改）
- **任务说明**（可选）

## 任务执行

创建后进入执行状态，实时展示进度。

## 任务记录

- 每次完成任务自动保存到历史
- 可查看结果详情
- 支持"重新执行"（复用参数再次采集）

# 采集引擎设计

## 1. 命令执行模型

### 三种命令类型

**类型A：简单命令**
直接执行，无依赖，无变量。
```json
{"cmd": "display ip interaccessce brief"}
```

**类型B：参数化命令（解决拼接问题）**
命令中使用 `{变量名}`，值来自场景输入参数或上一步结果。
```json
{
  "cmd": "display ip routing-table {target_network}",
  "params": ["target_network"]  // 场景输入参数
}
```
运行时：`display ip routing-table 10.0.0.0/24`

**类型C：派生命令（解决依赖关系）**
先执行父命令，解析输出得到一组值，再为每个值执行子命令。
```json
{
  "cmd": "display ip vpn",
  "type": "foreach",
  "parse": "regex: vpn-instance ([\\w-]+)",  // 提取VPN名称列表
  "foreach_var": "vpn_name",
  "sub_commands": [
    {"cmd": "display ip routing-table vpn-instance {vpn_name} {target_network}"}
  ]
}
```
执行流程：
```
→ display ip vpn
→ 解析出 vpn-a, vpn-b, vpn-c
→ display ip routing-table vpn-instance vpn-a {target_network}
→ display ip routing-table vpn-instance vpn-b {target_network}
→ display ip routing-table vpn-instance vpn-c {target_network}
```

## 2. 大结果处理（解决1M路由表）

### 策略
- **分块读取**：paramiko channel 边收边写，不全部放内存
- **流式写入**：直接写到临时文件，内存只保留最后100行用于进度判断
- **超时保护**：每条命令独立超时，长时间无输出自动中断
- **大小限制**：单条命令输出上限可配（默认50MB），超限截断并告警

### 采集进度
```
设备1/8: 采集命令1/3 (路由表检查) — VPN 2/5 — 已收 2.3MB
```

## 3. 设备级采集流程

```
连接SSH
  ↓ expect到设备
  ↓ 按顺序执行命令（同设备串行，不同设备并行）
  │
  ├── 简单命令 → 直接收结果 → 写入本地文件
  ├── 参数化命令 → 替换变量 → 执行 → 写入
  └── 派生命令 → 解析父结果 → foreach执行子命令 → 写入
  ↓
关闭SSH
  ↓
标记完成/失败
```

### 结果存储
```
tasks/
├── task_20260614_01/
│   ├── task.json           // 任务信息
│   ├── devices/
│   │   ├── 192.168.1.1/
│   │   │   ├── cmd_1_display_ip_routing.txt
│   │   │   ├── cmd_2_display_bgp_peer.txt
│   │   │   └── status.json    // 采集状态
│   │   └── 192.168.2.1/
│   │       └── ...
│   └── summary.json        // 汇总
```

## 4. 错误处理
- 单设备失败不影响其他设备
- 单命令失败不影响其他命令
- 超时、连接断开自动重试1次
- 所有错误记录到设备的状态文件中

## 5. 你可能还没考虑到的

### 命令超时
每条命令有独立超时，不是整台设备一个超时。路由表采集可能需要120s+，接口状态10s就够。

### 厂商差异
同一命令在不同厂商设备上格式不同。建议命令集支持 vendor 过滤：
```
命令集"路由表检查"：
  VendorA: display ip routing-table
  VendorB: show ip route
```

### 采集前检查
开始采集前先确认设备可达（ping或SSH预连），避免等半天发现设备不通。

### 采集中断恢复
如果采到一半中断（SSH断了），支持从中断点继续，而不是重头采全部。

### 敏感命令保护
write、delete、reset 等写操作命令不应出现在采集命令集中，加校验拦截。

# 设计方案优化补充

## 1. 配置存储：JSON + SQLite 双模式

### 现状问题
全部 JSON 文件存储，运行时频繁增删改查（SSH连接、设备、场景）效率低，并发写入可能丢数据。

### 优化方案
```
编辑期             运行时             导入导出
GUI编辑 ↔ SQLite ←→ 内存缓存    JSON文件 ←→ SQLite
          ↑                        ↕
     （持久化）               （备份/迁移）
```

SQLite 的好处：
- 事务支持，并发安全
- 按ID/名称查询快
- 版本管理方便
- 单文件数据库，部署简单

### 数据表设计

```sql
-- SSH连接
CREATE TABLE ssh_connections (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT UNIQUE NOT NULL,
    host TEXT NOT NULL,
    port INTEGER DEFAULT 22,
    username TEXT,
    key_path TEXT,
    key_password TEXT,  -- AES加密存储，密钥来自用户设定的主密码
    expect_flow TEXT,    -- JSON数组
    created_at TEXT,
    updated_at TEXT
);

-- Region+Section映射
CREATE TABLE region_mapping (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    region TEXT NOT NULL,
    section TEXT NOT NULL,  -- 支持通配符: all, az*
    ssh_connection_id INTEGER REFERENCES ssh_connections(id),
    is_deaccessult INTEGER DEFAULT 0,  -- 是否该Region的默认连接
    UNIQUE(region, section)
);

-- 设备组模板（全局）
CREATE TABLE device_group_templates (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT UNIQUE NOT NULL,
    section_pattern TEXT NOT NULL,
    role TEXT NOT NULL,
    description TEXT,
    created_at TEXT
);

-- 命令集
CREATE TABLE command_sets (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT UNIQUE NOT NULL,
    vendor TEXT,  -- VendorA/VendorB/null(通用)
    description TEXT,
    commands TEXT NOT NULL,  -- JSON数组
    created_at TEXT,
    updated_at TEXT
);

-- 场景模板
CREATE TABLE scene_templates (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    scene_type TEXT NOT NULL,
    description TEXT,
    analyzer_plugin TEXT NOT NULL,  -- 绑定的分析插件
    web_system TEXT DEFAULT '',     -- 关联Web系统: ''/ipam/cmdb/...
    version INTEGER DEFAULT 1,
    input_params TEXT,              -- JSON，场景输入参数定义
    sub_scenes TEXT,                -- JSON，子场景列表
    device_groups TEXT,             -- JSON，设备组定义（引用模板或内联）
    command_set_ids TEXT,           -- JSON，引用的命令集ID列表
    is_template INTEGER DEFAULT 0, -- 是否预置模板
    created_at TEXT,
    updated_at TEXT
);
```

### JSON ↔ SQLite 转换
- 导出：SQLite → JSON文件（密码字段清空）
- 导入：JSON文件 → SQLite（密码字段留空，手动补填）

## 2. 设备组模板 → 全局配置

### 现状问题
设备组模板定义在场景内部，多个场景共用同一模板（如`az_access`）时重复定义。

### 优化方案
设备组模板移到全局配置层，场景引用即可：

```
全局配置
├── SSH连接
├── Region+Section映射
├── 设备组模板 ← 新增
│   ├── az_access: section=az*, role=access
│   ├── Transit-Core: section=Transit*, role=core
│   └── ...
├── 命令集
└── 场景模板（引用上面的模板和命令集）
```

场景内也可定义私有设备组（不对外暴露），优先级高于全局模板。

## 3. 命令输出格式声明

### 现状问题
分析插件拿到的是原始文本，每个插件都要自己写文本解析器，重复劳动且易错。

### 优化方案
命令集支持声明输出格式，引擎自动解析为结构化数据：

```json
{
  "name": "路由表检查",
  "commands": [
    {
      "cmd": "display ip routing-table",
      "output_format": {
        "type": "table",
        "columns": ["network", "next_hop", "interaccessce", "protocol"],
        "rules": [
          {
            "skip_lines": ["^Total", "^-+", "^$"],     // 跳过表头、分隔线、空行
            "extract": "regex: (\\S+)\\s+(\\S+)\\s+(\\S+)\\s+(\\S+)"
          }
        ]
      }
    }
  ]
}
```

引擎执行后自动解析：
```python
原始输出 → 引擎自动解析
{
  "raw_file": "tasks/xxx/device/cmd_1.txt",  # 原始文本也保留
  "parsed": [
    {"network": "10.0.1.0/24", "next_hop": "10.0.0.1", "interaccessce": "G0/0/1", "protocol": "OSPF"},
    {"network": "10.0.2.0/24", "next_hop": "10.0.0.2", "interaccessce": "G0/0/2", "protocol": "Static"}
  ]
}
```

分析插件直接拿 parsed 数组做分析，不用碰文本。

支持的输出格式类型：
| 类型 | 说明 | 解析方式 |
|------|------|---------|
| table | 表格 | 正则逐行提取列 |
| key-value | 键值对 | 按分隔符拆分 |
| list | 列表 | 每行一个值 |
| raw | 原始文本 | 不做解析，插件自己处理 |

## 4. 任务取消机制

### 设计
- 任务执行中，用户可点"取消任务"
- 取消操作：
  1. 标记任务状态为"cancelling"
  2. 中断正在执行的采集线程
  3. 等待已完成的设备写入完毕
  4. 标记任务状态为 "cancelled"
  5. 已采集的设备结果保留，已有多少算多少

### 界面交互
- 任务列表 → 正在执行的任务 → 取消按钮
- 取消后显示：**已取消（已采集 5/8 台设备）**

## 5. 任务通知

### 触发时机
- 任务全部完成
- 任务部分失败（部分设备采集失败）
- 任务被取消

### 通知方式
- 任务列表状态更新（被动）
- 系统通知弹窗
- 可扩展：邮件通知（后续）

## 6. 场景版本锁定

### 规则
- 创建任务时，记录当前场景模板的版本号
- 任务始终使用创建时的版本进行分析
- 场景模板更新后，旧任务不受影响
- 重新执行任务时，询问用户：
  "上次使用场景 v2.0，当前已更新到 v3.0，使用新版本还是旧版本？"

### 数据结构
```sql
-- 任务表
CREATE TABLE tasks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT,
    scene_template_id INTEGER,
    scene_version INTEGER,        -- 创建时锁定的版本
    scene_snapshot TEXT,          -- 创建时的场景完整快照（JSON）
    region TEXT,
    status TEXT,                  -- pending/running/completed/accessiled/cancelled
    input_params TEXT,            -- JSON，用户填的输入参数
    device_list TEXT,             -- JSON，实际采集的设备列表
    created_at TEXT,
    completed_at TEXT
);
```

# 设计方案优化补充 v2 — 专家评审

## 7. SSH 连接复用（性能关键）

### 问题
同一个 Section 有多个设备，当前设计是每个设备独立 SSH 连接。
```
Section az1 有 3 台设备 → 开 3 次 SSH 到苏州堡垒机 → expect 跳 3 次
```
每次 SSH 握手 + bastion expect 耗时 3-5s，10 台设备就浪费 30-50s。

### 优化
**同 Section 的设备复用同一个 SSH 连接**，轮流 expect 跳转：
```
一次SSH连到苏州堡垒机
  → expect跳到设备A，采集，断开
  → expect跳到设备B，采集，断开
  → expect跳到设备C，采集，断开
  → 关闭SSH
```

实现：按 Section 分组，每个 Section 建立一个 SSH session，设备串行采集。

## 8. 并发控制

### 问题
多 Section 和多设备同时跑，没有限制可能导致：
- 大量 SSH 连接撑爆系统（ulimit限制）
- GUI 卡死
- 堡垒机侧可能限流

### 优化
```
全局并发上限配置
├── 最大并行 Section 数：3（默认）
├── 同 Section 内串行（复用SSH连接）
└── 最大总设备并发：10（默认）
```

任务执行时：
```
Section az1（设备A→B→C 串行）
        ↕ 并行
Section az2（设备D→E→F 串行）
        ↕ 并行
Section transit1（设备G 单台）
```

## 9. 历史对比 — 结果差异分析

### 问题
当前分析结果只针对单次任务。实际场景中常需要对比"上次结果 vs 这次结果"。
例如：上周查没冲突，这周突然冒出3个 → 应该自动标出来。

### 优化
- 每次任务结果保存时，自动与前一次同场景同Region的任务结果做**差异对比**
- 输出中标注：`新增冲突 2 个，已解决冲突 1 个`
- 界面展示：冲突列表按"新增/持续/已解决"分类

## 10. 日志与可观测性

### 问题
运行时出问题（SSH超时、解析失败、插件报错），用户只能靠猜。

### 优化
每层写结构化日志，统一查看：

```
logs/
├── app.log              # 应用层（任务创建/取消）
├── engine.log           # 采集引擎（SSH连接/命令执行）
│   └── 每一行：时间 | 任务ID | 设备IP | 命令 | 状态 | 耗时
├── analyzer.log         # 分析插件
└── db.log               # 数据库操作
```

关键指标打点：
- 任务平均耗时
- SSH连接成功率
- 单命令平均执行时间
- 分析插件执行时间

## 11. SSH 连接健康检查

### 问题
配置了 SSH 连接但堡垒机当前不可用（网络/维护），任务跑一半才发现，浪费时间。

### 优化
- 全局配置中有**连接测试**按钮
- 配置保存时可选"测试连接"
- 任务开始前，对所有涉及的 SSH 连接执行预检（ping + SSH端口通）
- 预检不通过 → 直接提示，不进采集流程

## 12. 数据库迁移

### 问题
工具版本升级，SQLite 表结构变化 → 旧数据库不兼容。

### 优化
- 数据库中记录 schema_version
- 工具启动时检查版本号
- 旧版本自动执行迁移脚本（新增字段/表）
- 迁移失败回滚并告警

## 13. 分析插件沙盒

### 问题
第三方分析插件 crash（死循环、内存泄漏） → 拖垮整个工具。

### 优化
- 插件在独立子进程中执行
- 设置超时（默认 60s）和内存上限（默认 256MB）
- 插件崩溃 → 仅该场景分析失败，不影响其他
- 插件输出 JSON 合法性校验

## 14. 定时任务（巡检模式）

### 场景
每天凌晨自动跑一遍"IP冲突检测"，有冲突自动通知。

### 设计
- 任务创建时可设为"定时任务"
- 支持 cron 表达式（如 `0 2 * * *` 每天凌晨2点）
- 执行结果静默保存，只有"有问题"才通知
- 连续 N 次正常 → 降低通知频率
